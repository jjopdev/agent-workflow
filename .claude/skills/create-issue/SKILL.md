# Create Issue ‚Äî From Analysis to Tracking

After you have analyzed a problem and generated a plan, this skill documents everything and creates the tracking artifacts.

**Input:** $ARGUMENTS

## Context

This is used AFTER Claude has already:
1. Received a problem description from the user (client error, feature request, etc.)
2. Investigated the codebase and found the relevant files
3. Generated a plan

Now we need to document it and create the issue for tracking.

## Process

### 1. Summarize the analysis
From the conversation context, extract:
- **Problem description** ‚Äî what the client reported
- **Root cause / scope** ‚Äî what Claude found in the codebase
- **Affected files** ‚Äî specific paths and functions
- **Plan** ‚Äî the steps to fix/implement
- **Acceptance criteria** ‚Äî how to verify it's done

### 2. Classify the task
| Type | Branch prefix | Label | Priority |
|------|--------------|-------|----------|
| Bug (production down) | `fix/` | `bug` | Critical |
| Bug (functional) | `fix/` | `bug` | High/Medium |
| Feature (new) | `feature/` | `enhancement` | Medium |
| Migration | `migration/` | `migration` | Medium |
| Improvement | `improve/` | `improvement` | Low/Medium |
| Refactor | `refactor/` | `refactor` | Low |

Estimate size: **S** (<4h), **M** (1-2d), **L** (3-5d), **XL** (1+ week)

### 3. Create Notion page
Use MCP to create a page in the project's Notion workspace:

Search for the project board first:
```
notion-search: query for the project name or "Board de Tareas"
```

Create a page with:
- **Title**: `[Type] Short description`
- **Status**: "En An√°lisis" or "Ready"
- **Priority**: Based on classification
- **Size**: S/M/L/XL
- **Content**: Full analysis, plan, affected files, acceptance criteria

### 4. Create GitHub Issue
```bash
gh issue create --title "<type>: <short description>" --body "$(cat <<'EOF'
## Context
[What the client reported and why this matters]

## Analysis
[What was found ‚Äî root cause for bugs, scope for features]

## Current Behavior
[What happens now]

## Expected Behavior
[What should happen after the fix/implementation]

## Technical Plan
### Files to modify
- `path/to/file.ts:line` ‚Äî [what to change]
- `path/to/file.ts:line` ‚Äî [what to change]

### Steps
1. [Step 1]
2. [Step 2]
3. [...]

### Acceptance Criteria
- [ ] [Criterion 1]
- [ ] [Criterion 2]
- [ ] [Criterion 3]

### Out of Scope
- [What NOT to touch]

## Branch
`<prefix>/<issue-number>-<short-description>`

## Size: [S|M|L|XL]

---
> Plan generated by Claude Code Agent Workflow
EOF
)" --label "<label>"
```

### 5. Link Notion ‚Üî GitHub
- Update the Notion page with the GitHub Issue URL
- Report both links to the user

### 6. Ask the user
After creating both artifacts, ask:

> Issue #XX created. Notion updated.
> Do you want to:
> 1. **Execute it yourself** ‚Äî I'll run the full workflow (Plan‚ÜíImplement‚ÜíTest‚ÜíReview‚ÜíSecurity)
> 2. **Delegate to a developer** ‚Äî The issue is ready, assign it to a dev

## Rules
- This skill assumes Claude has ALREADY analyzed the problem in the current conversation
- If no analysis exists yet, tell the user to describe the problem first so you can investigate
- The plan must be specific enough that a developer can execute WITHOUT asking questions
- Include actual file paths and function names from the codebase ‚Äî not vague descriptions
- If the task is XL, suggest splitting into multiple issues
- Always create BOTH Notion page AND GitHub Issue ‚Äî that's the point of this skill
- The branch name uses the GitHub Issue number, not a Notion task ID
